3. Ajouter l'en-tête `Strict-Transport-Security` (HSTS) dans `public/_headers`

## Findings détaillés

### 🟡 [MOYENNE] Formulaire de contact sans rate limiting ni anti-bot fort

**Axe** : 9 — Rate limiting & abuse prevention
**Fichier** : `functions/api/contact.ts:18-61`
**Description** : L'endpoint `POST /api/contact` n'applique qu'un honeypot (`bot-field`).
Aucun rate limiting par IP, aucun CSRF token, aucun CAPTCHA/Turnstile. La validation
des champs (longueur, regex email) est correcte mais ne limite pas la fréquence des requêtes.
**Preuve** :
```ts
if (form.get("bot-field")) return text("OK", 200);   // seule protection anti-bot
// ... aucune vérification de fréquence avant le fetch Resend
const res = await fetch("https://api.resend.com/emails", { ... });
```
**Impact** : Un attaquant qui contourne le honeypot (trivial — il suffit de ne pas remplir
`bot-field`) peut automatiser des milliers de POST → (a) email-bombing de la boîte `CONTACT_TO`,
(b) épuisement du quota Resend et coûts associés.
**Remédiation** (par ordre de préférence, même écosystème Cloudflare) :
- **Cloudflare Turnstile** (gratuit) : widget côté `ContactModal` + vérification du token serveur dans `contact.ts`
- **Cloudflare Rate Limiting Rules** (dashboard) sur le path `/api/contact`
- Rate limit applicatif via **Workers KV** (compteur par IP `request.headers.get("CF-Connecting-IP")`)
**Ref** : CWE-770 (Allocation of Resources Without Limits), OWASP API4:2023 (Unrestricted Resource Consumption)

### 🔵 [BASSE] En-tête HSTS (Strict-Transport-Security) absent

**Axe** : 6 — Configuration serveur & déploiement
**Fichier** : `public/_headers` (absent)
**Description** : Aucun `Strict-Transport-Security` n'est défini. Cloudflare force HTTPS,
mais sans HSTS un client peut être vulnérable au SSL-stripping lors du tout premier accès
(avant redirection) ou via un proxy MITM.
**Impact** : Fenêtre étroite de downgrade HTTPS→HTTP au premier contact. Faible pour un
site sans authentification, mais trivial à corriger.
**Remédiation** : ajouter dans `public/_headers` sous le bloc `/*` :
```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```
> Note : vérifier au préalable que `drouet.io` n'est pas déjà couvert par une règle HSTS
> au niveau zone Cloudflare (dashboard → SSL/TLS → Edge Certificates), pour éviter un doublon.
**Ref** : CWE-319 (Cleartext Transmission), OWASP A05:2021

### 🔵 [BASSE] CSP : `'unsafe-inline'` sur `script-src`

**Axe** : 6 — Configuration serveur & déploiement
**Fichier** : `src/layouts/BaseLayout.astro:145`, `public/_headers:83`
**Description** : La CSP autorise `script-src 'self' 'unsafe-inline'`. `'unsafe-inline'`
neutralise la principale protection CSP contre le XSS injecté.
**Impact** : Faible en l'état — le site est statique et ne rend aucun input utilisateur
externe en HTML (cf. finding INFO ci-dessous). Mais en cas de future faille d'injection,
la CSP n'offrirait pas de filet de sécurité sur les scripts.
**Remédiation** : remplacer `'unsafe-inline'` par des hashes CSP (`'sha256-...'`) pour les
scripts inline d'Astro (`is:inline`, JSON-LD). Astro peut générer les hashes via l'intégration
`@astrojs/csp` (expérimental) ou calcul manuel. Effort moyen pour un gain marginal sur ce site.
**Ref** : CWE-1021, OWASP A05:2021

### ⚪ [INFO] `set:html` multiples sur contenu de confiance

**Axe** : 2 — Injection (XSS)
**Fichiers** : `src/components/cv/CVCard.astro:25`, `src/components/ExperienceCard.astro:66,87,90,93,103,105`, `src/pages/index.astro:52,62,84,106`
**Description** : Nombreux `set:html` (équivalent `innerHTML`). Traçage des sources :
- `company`, `role`, `period`, `achievement`, `item`, `contactItem`, `interest` → issus du
  parsing de `src/content/cv/cv.md` (**contenu propriétaire**, pas d'input externe)
- `iconSVG` → SVG fetché depuis `api.iconify.design` **au build SSG** (figé dans le HTML)
- JSON-LD (`BaseLayout.astro:74`) → `JSON.stringify(...)` (échappement automatique)

**Impact** : Aucun vecteur XSS exploitable par un tiers — le site ne rend aucun input
utilisateur en HTML. Le seul input externe (formulaire de contact) part en email `text/plain`,
jamais réinjecté dans le DOM. Risque résiduel théorique : supply-chain iconify (SVG malveillant
au build) — atténué par le fait que `iconName` n'est pas user-controlled.
**Remédiation** : aucune action requise. Surveiller si un jour du contenu externe/dynamique
devait alimenter ces composants → sanitiser alors (DOMPurify).
**Ref** : CWE-79

## Points positifs notés

- ✅ `functions/api/contact.ts` : validation email (regex), troncature des champs (`clip`),
  honeypot, secrets en variables d'environnement, fetch vers URL Resend **hardcodée** (pas de SSRF).
  `reply_to`/`subject` passés en **JSON** à l'API Resend → pas d'injection d'en-tête SMTP.
- ✅ `functions/_middleware.ts` : `ROUTE_TO_MARKDOWN` est une **allowlist statique** → pas de
  path traversal ; `assets.fetch` borné à l'origine → pas de SSRF.
- ✅ Secrets : aucun secret hardcodé, `.env` + `.env.production` gitignored, historique git propre.
- ✅ En-têtes de sécurité présents : `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`,
  `Referrer-Policy`, `Permissions-Policy`, CSP complète. X-Frame-Options + Permissions-Policy
  injectés via middleware (CF Pages ignore ces 2 dans `_headers`).
- ✅ Migration récente vers `font-src 'self'` (auto-hébergement polices) → CSP plus stricte,
  suppression des origines tierces `fonts.googleapis.com`/`fonts.gstatic.com`.
